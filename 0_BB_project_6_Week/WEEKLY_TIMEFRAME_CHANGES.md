# Изменения для перехода на недельный таймфрейм

## Дата: 2024
## Описание: Конвертация системы с дневного (D1) на недельный (W1) таймфрейм

---

## 📋 Изменённые файлы

### 1. `data_loader.py`
**Изменения:**
- `find_earliest_available_date()`: 
  - Заменён интервал свечей с `CANDLE_INTERVAL_DAY` на `CANDLE_INTERVAL_WEEK`
  - Изменён шаг бинарного поиска с `timedelta(days=1)` на `timedelta(days=7)`

- `get_candles()`:
  - Загружает недельные свечи вместо дневных
  - Интервал: `CandleInterval.CANDLE_INTERVAL_WEEK`

- `calculate_bollinger_bands()`:
  - Обновлена документация: окно SMA теперь в неделях (20 недель ≈ 5 месяцев)

**Влияние:** Теперь данные загружаются и сохраняются в БД как недельные свечи.

---

### 2. `signals_processor.py`
**Изменения:**
- Константа `N`: 
  - Комментарий изменён с "дней" на "недель"
  - Значение осталось 5 (теперь это 5 недель истории для проверки сигналов)

- `get_last_n_days()`:
  - Переименован комментарий: "Получает последние N недель котировок"
  - Логика не изменилась, но теперь работает с недельными данными

- `find_trend_change()`:
  - Обновлена документация: указано, что функция работает на недельном таймфрейме

- `check_signals()`:
  - Изменён вывод в логах: "последняя неделя" вместо "последняя дата"

**Влияние:** Сигналы теперь генерируются на основе недельных данных. Период SMA(20) = 20 недель (~5 месяцев).

---

### 3. `trader_executor.py`
**Изменения:**
- Функция `get_last_n_days()` переименована в `get_last_n_weeks()`:
  - Обновлены комментарии и названия переменных
  - `last_candle_date` → `last_week_date`

- Основной цикл исполнения сделок:
  - Вызывает `get_last_n_weeks()` вместо `get_last_n_days()`
  - Проверка сигналов использует дату последней недели

**Влияние:** Исполнение сделок происходит на основе недельных сигналов.

---

### 4. `config.py`
**Изменения:**
- Константа `N_DAYS` переименована в `N_WEEKS = 120`
  - 120 недель ≈ 2.3 года исторических данных для тестирования

- `BOLLINGER_CONFIG`:
  - Обновлены комментарии: "Период для SMA (20 недель)"
  
- `API_LIMITS`:
  - `candles_per_request`: увеличено с 30 до 365 (недель за запрос)
  - Это позволяет загружать ~7 лет данных за один запрос

**Влияние:** Конфигурация адаптирована для недельного таймфрейма.

---

### 5. `main.py`
**Изменения:**
- Функция `data_is_ready()`:
  - Проверяет наличие данных за предыдущую неделю (`INTERVAL '7 days'`)
  - Вместо проверки вчерашнего дня

**Влияние:** Оркестратор корректно ожидает загрузки недельных данных.

## 🔍 Ключевые отличия недельного таймфрейма от дневного

| Параметр | Дневной (D1) | Недельный (W1) |
|----------|--------------|----------------|
| Период SMA(20) | 20 дней (~1 месяц) | 20 недель (~5 месяцев) |
| Чувствительность | Высокая | Низкая (сглажена) |
| Количество сигналов | Много | Меньше, но качественнее |
| Ложные срабатывания | Чаще | Реже |
| Требуемый объём данных | 120 дней | 120 недель (2.3 года) |
| Время удержания позиции | Дни/недели | Недели/месяцы |

## ⚠️ Важные замечания

1. **Период индикаторов**: Полосы Боллинджера с периодом 20 на недельном графике означают усреднение за ~5 месяцев, что делает стратегию менее чувствительной к краткосрочным колебаниям.

2. **Исторические данные**: Для корректной работы требуется загрузить минимум 120 недель (~2.3 года) исторических данных для каждого тикера.

3. **Частота запуска**: 
   - **Рекомендуется**: Запускать раз в неделю (например, в понедельник после закрытия предыдущей недели)
   - **Допустимо**: Ежедневный запуск, но сигналы будут формироваться только по факту закрытия недельной свечи

4. **Тестирование**: Перед реальной торговлей рекомендуется протестировать стратегию на исторических данных (backtest) с учётом недельного таймфрейма.


   ```bash

   ```

   - Добавить запуск по расписанию (cron/systemd)
   - Рекомендуется: раз в неделю после закрытия торговой сессии
